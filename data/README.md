# Cursor Usage Data Runbook

This directory stores data generated by the Cursor usage ingestion workflow.

## What this subsystem does

The script at `scripts/fetch_cursor_usage.py` runs hourly via `.github/workflows/cursor-usage.yml` and:

1. Reads `CURSOR_API_KEY` from environment/secrets.
2. Pulls usage data from Cursor APIs for the current UTC day.
3. Appends snapshot rows to SQLite (`cursor_usage.db`).
4. Writes a summary JSON (`cursor_usage_latest.json`) used by README badges.

## Files and responsibilities

- `cursor_usage.db`: SQLite database storing historical snapshot rows.
- `cursor_usage_latest.json`: Latest summary used by dynamic README badge(s).
- `schema.sql`: Schema and indexes for `usage_snapshots`.
- `../scripts/fetch_cursor_usage.py`: Ingestion and summary generation logic.
- `../.github/workflows/cursor-usage.yml`: Hourly automation that runs ingestion and commits `data/` changes.

## Data model

Table: `usage_snapshots` (defined in `schema.sql`)

- `recorded_at` (`TEXT`): ISO hour bucket (`YYYY-MM-DDTHH:00:00Z`).
- `metric_type` (`TEXT`): One of `agent_edits`, `tabs`, `ai_commits` (plus future values if added).
- `project` (`TEXT`): Repo name for project allocation (primarily from AI commit records).
- `user_id` (`TEXT`): Optional user identifier/email from source payload.
- `value_json` (`TEXT`): Raw API payload row for traceability.
- `source` (`TEXT`): Source system label (`cursor_analytics`, `cursor_ai_code`, etc.).

Indexes exist on `recorded_at`, `project`, and `metric_type`.

## Upstream APIs used

All endpoints are called with `startDate=<today>` and `endDate=<today>` in UTC:

- `/analytics/team/agent-edits`
- `/analytics/team/tabs`
- `/analytics/ai-code/commits?pageSize=100`

Authentication uses HTTP Basic auth with `CURSOR_API_KEY`.

## Summary JSON contract (`cursor_usage_latest.json`)

### Successful run

```json
{
  "display_value": "6",
  "recorded_at": "2026-04-06T16:00:00Z",
  "snapshots_today": 6,
  "last_7d": {"agent_edits": 2, "tabs": 1, "ai_commits": 3},
  "by_project": {"repo-a": 2, "repo-b": 1}
}
```

### Missing API key

```json
{
  "display_value": "CURSOR_API_KEY not set",
  "snapshots_today": 0
}
```

### API/plan error

```json
{
  "display_value": "NEED TO UPGRADE TO ENTERPRISE PLAN TO RETURN VALUE",
  "snapshots_today": 0,
  "recorded_at": "2026-04-06T16:00:00Z",
  "last_7d": {},
  "by_project": {}
}
```

## Setup

1. Add repository secret `CURSOR_API_KEY` (Cursor Admin API key).
2. Ensure Cursor plan supports Analytics + AI Code Tracking APIs (Enterprise requirement).
3. Confirm workflow permissions allow committing `data/` updates.

## Local usage

Run ingestion manually:

```bash
export CURSOR_API_KEY=your_key_here
python scripts/fetch_cursor_usage.py
```

Run tests for this subsystem:

```bash
python -m unittest tests/test_fetch_cursor_usage.py
```

## Workflow behavior and constraints

- Schedule: hourly (`0 * * * *`) plus manual `workflow_dispatch`.
- Workflow step uses `continue-on-error: true` for script execution.
- Script exits successfully (`0`) even when key is missing or API returns Enterprise/auth errors; it writes a summary message instead of failing hard.
- On first API error, remaining API calls are skipped for that run.
- `display_value` is the number of rows inserted in the current run, not total token consumption.

## Troubleshooting

### Badge shows `CURSOR_API_KEY not set`

- Verify repository secret exists and is named exactly `CURSOR_API_KEY`.
- Re-run workflow manually and inspect the fetch step logs.

### Badge shows Enterprise upgrade message

- API key is valid but account/plan does not expose required endpoints.
- Upgrade/enable required Cursor API access.

### `by_project` is empty

- Expected if no AI commit rows were returned for the selected day.
- Also expected on API error runs (summary intentionally clears `by_project`).

### Database exists but counts look low

- Ingestion is day-scoped (`startDate=today`, `endDate=today`), so results depend on same-day activity.
- Check query time windows and UTC boundaries.

## Query examples

```sql
-- Metric counts in last 7 days
SELECT metric_type, COUNT(*) AS rows
FROM usage_snapshots
WHERE recorded_at >= datetime('now', '-7 days')
GROUP BY metric_type
ORDER BY rows DESC;
```

```sql
-- Project allocation in last 7 days
SELECT project, COUNT(*) AS rows
FROM usage_snapshots
WHERE recorded_at >= datetime('now', '-7 days')
  AND project IS NOT NULL
GROUP BY project
ORDER BY rows DESC
LIMIT 20;
```
